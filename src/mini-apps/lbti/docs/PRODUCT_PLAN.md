# LBTI 프로젝트 기획서

## 1. 제품 정의

**제품명:** LBTI: 로스쿨생 MBTI 테스트  
**형태:** Law Solver 안에서 실행되는 오락·자기점검형 미니 앱  
**핵심 질문:** 시간과 기억력이 부족한 수험 상황에서 나는 어떤 공부 결정을 먼저 내리는가?

LBTI는 타고난 성격이나 학업능력을 측정하지 않는다. 교수저·강사저·암기장, 풀커버·압축, 회독·현출, 루틴·마감처럼 로스쿨 수험생활에서 반복되는 선택을 네 축으로 재구성해 현재의 공부 경향을 재미있게 보여주는 콘텐츠다.

## 2. 기획 배경

로스쿨 공부법은 한 줄의 서열로 설명하기 어렵다. 교수저를 보면서 기출 부분만 추리는 사람도 있고, 강사저에 OX·사례집·최신판례를 계속 붙이는 사람도 있다. 암기장을 많이 읽는 것과 실제 답안에서 현출하는 것도 별개의 선택이며, 매일 누적하는 방식과 마감 직전 몰입하는 방식도 상황에 따라 섞인다.

따라서 LBTI는 다음 네 가지 의사결정을 분리한다.

1. `P–T`: 최종 기준을 원전과 수험자료 중 어디에 두는가
2. `W–C`: 공부 범위를 넓게 가져갈지 핵심으로 압축할지
3. `R–O`: 기억이 불안할 때 다시 읽을지 직접 꺼내볼지
4. `S–D`: 진도를 매일 누적할지 마감 중심으로 압축할지

이 기획은 커뮤니티에서 반복되는 화제를 콘텐츠로 재구성한 것이며, 로스쿨생 전체에 대한 통계적 일반화가 아니다.

## 3. 목표와 비목표

### 목표

- 로스쿨생이 자신의 공부 선택을 가볍게 돌아보게 한다.
- 각 결과가 “내 얘기 같다”는 현실감과 공유하고 싶은 재미를 갖게 한다.
- 특정 교재나 공부법을 정답으로 제시하지 않고 각 방식의 효용과 과몰입 위험을 함께 보여준다.
- Law Solver 랜딩과 자연스럽게 연결되는 완결된 미니 앱 경험을 만든다.

### 비목표

- 심리검사, 성격장애 또는 정신건강 진단
- 성적·변호사시험 합격 가능성 예측
- 과학적 타당성이 검증된 표준화 검사처럼 보이게 하는 것
- 특정 교수, 강사, 교재 또는 커뮤니티 이용자를 평가하는 것
- 사용자 답안이나 결과의 서버 저장·수집

## 4. 대상 사용자

- 로스쿨 재학생과 변호사시험 수험생
- 입학 전 로스쿨 공부문화를 가볍게 이해하고 싶은 사람
- 자신의 공부법을 동기와 비교하거나 결과 카드를 공유하고 싶은 사용자

주요 문구는 로스쿨 용어를 아는 사용자에게 현실감 있게 작성하되, 특정 커뮤니티 내부 표현을 모르는 사람도 결과의 뜻을 이해할 수 있어야 한다.

## 5. 테스트 모델

### 지표

지표와 유형의 단일 콘텐츠 원본은 `../data/lbti-framework.json`이다.

| 순서 | 축 | 왼쪽 | 오른쪽 | 측정 대상 |
| --- | --- | --- | --- | --- |
| 1 | P–T | P: 원전파 | T: 수험파 | 최종 판단기준 |
| 2 | W–C | W: 풀커버형 | C: 압축형 | 학습 범위 |
| 3 | R–O | R: 회독형 | O: 현출형 | 기억 회복 방식 |
| 4 | S–D | S: 누적형 | D: 마감형 | 시간 운영 |

### 확정 문항 구조

- 축당 7개, 총 28개
- 한 문항은 한 축만 측정
- `나는 … 한다` 형식으로 구체적인 수험 행동을 제시
- `매우 그렇다`, `그렇다`, `그렇지 않다`, `전혀 그렇지 않다`의 강제 4단계 척도 사용
- 문항 순서는 축별로 묶지 않고 섞어서 노출
- 각 축의 양쪽 pole을 지지하는 진술문을 섞어 동의 응답 습관을 완화
- 답을 바꿀 수 있도록 이전 문항 이동 제공

운영 문항과 표시 순서는 `../data/questions.ko.json`을 단일 원본으로 사용한다.

### 채점 방식

응답 weight는 `매우 그렇다=3`, `그렇다=2`, `그렇지 않다=1`, `전혀 그렇지 않다=0`이다. 선택한 weight는 문항의 `agreement_pole`에 더하고, `3-weight`는 같은 축의 반대 pole에 더한다.

축당 7문항이므로 양쪽 pole 점수의 합은 항상 21이다. 합계가 홀수이므로 동점 처리 규칙 없이 더 높은 pole을 선택할 수 있다. 결과 코드는 `P/T → W/C → R/O → S/D` 순서로 조합한다.

원점수와 결과 코드 계산은 구현하기 단순하지만, 공개 전에는 문항 간 의미 중복과 특정 pole을 더 바람직하게 보이게 하는 표현을 검토한다. 중립 선택지는 사용하지 않으며 축별 점수를 결과 화면에서 숫자나 비율로 보여줄지는 별도로 결정한다.

## 6. 사용자 흐름

1. `/apps`에서 LBTI 카드를 선택한다.
2. 소개 화면에서 테스트 성격을 설명하는 짧은 안내를 확인한다.
3. `테스트 시작`을 누른다.
4. 28개 상황형 진술문에 4단계 척도로 답한다.
5. 제출 전 미응답 문항이 있으면 안내한다.
6. 네 축의 점수로 4자 코드를 계산한다.
7. 결과 화면에서 유형 콘텐츠와 축별 경향을 확인한다.
8. 결과 카드를 이미지 또는 링크 형태로 공유하거나 다시 테스트한다.

브라우저 뒤로가기·새로고침 정책, 답변 임시저장 여부와 공유 방식은 상세 구현 명세에서 확정한다.

## 7. 화면 범위

### 소개 화면

- LBTI 제목과 한 문장 설명
- 예상 소요시간과 28문항 안내
- 오락·자기점검 콘텐츠라는 한 문장 안내. 별도 경고 모달이나 반복 배너는 사용하지 않음
- 테스트 시작 CTA

### 질문 화면

- 현재 문항 번호와 진행률
- 수험 행동 진술문과 4단계 응답 선택지
- 이전/다음 이동
- 선택 여부가 색상 외에도 텍스트·테두리로 구분되는 접근성 상태

### 결과 화면

1. 4자 코드와 유형명
2. 대표 대사
3. 책상 위 풍경
4. 시험 30일 전 행동
5. 잘 맞는 공부법
6. 과몰입 시 나타나는 모습
7. 상극 유형과 스터디 궁합
8. 필요하면 독고다이·스터디장 등 보조 배지
9. 다시 하기와 공유

모든 유형의 상세 결과 콘텐츠는 아직 미확정이다. 현재 JSON의 `description`은 목록과 기획 검토용 요약이며 완성 결과문으로 간주하지 않는다.

## 8. 콘텐츠 원칙

- 모든 유형에 매력과 위험을 함께 둔다.
- `P`, `W`, `R`, `S` 또는 반대 pole을 정답처럼 표현하지 않는다.
- “게으르다”, “머리가 나쁘다”, “합격한다/떨어진다”처럼 사람의 능력과 결과를 단정하지 않는다.
- 교재·강사 실명은 운영 문항과 결과에서 원칙적으로 사용하지 않는다.
- 유형명과 유머는 별도 해명 없이도 실제 사람이나 집단을 비하하지 않는 표현을 사용한다.

세부 작성 기준은 `CONTENT_GUIDE.md`를 따른다.

## 9. 데이터와 개인정보

초기 버전은 서버 없이 브라우저에서 동작한다.

- 질문 답변과 결과를 GA4에 보내지 않는다.
- 4자 유형, 축별 점수, 진행률, 소요시간도 GA4에 보내지 않는다.
- 임시저장이 필요하면 `law-solver-mini-app:lbti:v1` namespace를 사용한다.
- 저장을 도입할 경우 데이터 타입, 만료/초기화 방식과 migration 테스트를 먼저 정의한다.
- LBTI 데이터는 기존 `law-solver-storage`와 Law Solver 전체 백업에 자동으로 섞지 않는다.

## 10. 접근성과 반응형

- 모바일을 기본으로 질문과 결과를 읽기 쉽게 구성한다.
- 4단계 응답 선택지는 키보드와 터치로 모두 조작할 수 있어야 한다.
- 색상만으로 선택 상태나 축 결과를 전달하지 않는다.
- 긴 한국어 문장은 `word-break: keep-all`과 충분한 행간을 사용한다.
- 다크 모드와 사용자의 Law Solver 글꼴 설정을 유지한다.
- 모션은 `prefers-reduced-motion`을 존중한다.

## 11. 분석 원칙

새 이벤트는 상세 구현 명세에서 별도 승인한 뒤 추가한다. 추가할 경우 이벤트 이름과 허용 파라미터를 `src/lib/analytics.ts`에 정적으로 제한하고 README, AGENTS, 개인정보처리방침을 함께 갱신한다.

허용 가능성이 있는 것은 LBTI 소개 진입·테스트 시작·완료처럼 내용과 성과를 담지 않는 최소 행동뿐이다. 답변, 결과 코드, 축 점수, 문항 번호, 소요시간과 내부 ID는 전송하지 않는다.

## 12. 출시 단계

### 단계 0 — 콘텐츠 기반 구축 (현재)

- 네 축과 16개 유형 JSON
- 프로젝트 기획서와 콘텐츠 가이드
- 출처와 안전 문구 정리

### 단계 1 — 상세 명세 확정

- 운영 문항 28개와 채점 설정
- 16개 상세 결과 콘텐츠
- 채점·재응시·임시저장·공유 정책
- 화면 와이어프레임과 카피

### 단계 2 — 핵심 구현

- 질문/채점 순수 로직과 단위 테스트
- 소개·질문·결과 화면
- `/apps/lbti` 라우팅과 manifest 상태 변경

### 단계 3 — 검수와 출시

- 문항 방향·유형 조합 자동 검증
- 모바일·다크 모드·키보드 접근성 확인
- 밈과 면책문구 콘텐츠 검수
- `npm test`, `npm run build`, SPA 새로고침 확인

## 13. 상세 명세에서 결정할 항목

- LBTI 영문 풀네임을 별도로 둘지 여부
- 운영 문항의 공개 전 편집·편향 검수 결과
- 답변 임시저장 및 새로고침 복구 여부
- 결과의 축별 점수를 숫자·비율로 보여줄지 여부
- 보조 배지의 산출 방식과 운영 여부
- 상극 유형과 스터디 궁합의 계산·서술 기준
- 공유 카드 생성 방식과 공유 URL에 포함할 정보
- 재응시 시 이전 결과를 보관할지 여부
